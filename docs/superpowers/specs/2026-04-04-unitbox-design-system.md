# Unitbox Design System (UDS) — Design Spec

> **Goal:** A living, AI-native design system that lets the team skip manual design and go straight to code — with consistent, scalable, customizable UI across all projects.

## Architecture: 3 Layers

### Layer 1: Knowledge (Claude Skill)

**Location:** `~/.claude/skills/unitbox-ds/` (git-synced across all team machines)

**Purpose:** Tells Claude *how to design* — patterns, anti-patterns, composition rules. Claude reads this skill automatically when doing any UI work.

**Structure:**
```
unitbox-ds/
  SKILL.md                      # Orchestrator: triggers on any UI work
  references/
    design-tokens.md            # Colors, typography, shadows, spacing
    component-rules.md          # Per-component radius, height, variant rules
    page-patterns.md            # 10+ page layout blueprints (JSX skeletons)
    anti-patterns.md            # What NEVER to do (40+ rules)
    list-patterns.md            # List row, avatar, action button patterns
    micro-components.md         # Small reusable atoms
    landing-patterns.md         # Marketing/landing page patterns
    ingestion-log.md            # Log: which dump → which patterns extracted
  scripts/
    parse-dump.py               # Parses DOM dump JSON → extracts tokens/patterns
    audit-tokens.sh             # Validates skill ↔ registry alignment
```

**Key rule:** The skill contains NO code — only markdown knowledge. Code lives in Layer 2.

### Layer 2: Code (shadcn Custom Registry)

**Location:** `unitbox-registry/` git repo → deployed on Vercel

**Purpose:** Installable code — tokens, overridden components, composed blocks. Any project installs with `npx shadcn add @unitbox/base`.

**Structure:**
```
unitbox-registry/
  registry.json                 # Manifest of all items
  registry/
    base/                       # registry:base — full design foundation
    font-inter/                 # registry:font — Inter font config
    ui/                         # registry:ui — overridden shadcn components
      button.tsx
      input.tsx
      badge.tsx
      card.tsx
      select.tsx
      dialog.tsx
      ... (all 20 overridden components)
    components/                 # registry:component — composed widgets
      filter-bar.tsx
      status-badge.tsx
      compact-list-row.tsx
      usage-meter.tsx
      announcement-banner.tsx
    blocks/                     # registry:block — page-level layouts
      data-table-page.tsx
      settings-page.tsx
      dashboard-layout.tsx
      card-grid-page.tsx
  public/r/                     # Auto-generated by `shadcn build`
```

**Registry protocol:** Uses official shadcn registry JSON schema. Each item declares its files, npm dependencies, and registry dependencies.

### Layer 3: Project (Consumer)

**Any Next.js project** that uses UDS.

**Setup (one-time):**
```bash
npx shadcn add @unitbox/base          # tokens + font + globals
npx shadcn add @unitbox/button        # overridden button (or bulk add)
npx shadcn add @unitbox/data-table    # page block
```

**Configuration in `components.json`:**
```json
{
  "registries": {
    "@unitbox": "https://registry.unitbox.dev/r/{name}.json"
  }
}
```

**Per-project customization** — override CSS variables in `globals.css`:
```css
/* Nexa CRM — blue brand */
:root {
  --primary: rgb(59, 130, 246);
  --radius-field: 0.5rem;
}
```

## @unitbox/base — What It Installs

The `registry:base` item is the foundation. It installs:

**CSS Variables (light + dark):**
```
Colors:     --background, --foreground, --primary, --secondary, --muted, --accent,
            --destructive, --border, --input, --ring, --popover, --card
            + sidebar variants, chart-1 through chart-5

Shadows:    --shadow-card, --shadow-card-hover, --shadow-modal,
            --shadow-elevation-sm, --shadow-popover

Radii:      --radius-field (12px), --radius-control (10px), --radius-menu (10px),
            --radius-menu-item (8px), --radius-card (16px), --radius-dialog (24px),
            --radius-pill (9999px)
```

**Font:** Inter via `registry:font` dependency

**Tailwind @theme extensions:** Semantic radius tokens that generate `rounded-field`, `rounded-control`, etc.

**@layer base rules:** `border-border`, `bg-background text-foreground`

## Semantic Token System

Components reference tokens, not hardcoded values:

| Token | Tailwind Class | Value | Components |
|-------|---------------|-------|------------|
| `--radius-field` | `rounded-field` | 12px | button, input, textarea, toggle, skeleton |
| `--radius-control` | `rounded-control` | 10px | select trigger, combobox trigger |
| `--radius-menu` | `rounded-menu` | 10px | dropdown/select/popover containers |
| `--radius-menu-item` | `rounded-menu-item` | 8px | items inside menus |
| `--radius-card` | `rounded-card` | 16px | cards |
| `--radius-dialog` | `rounded-dialog` | 24px | dialogs, modals |
| `--radius-pill` | `rounded-pill` | 9999px | badges, avatars, banners |

**Radius matching rule:** `inner_radius = outer_radius - padding`. Menu items (8px) inside menu containers (10px) with p-1 (4px).

**To change all button radii across all projects:** edit `--radius-field` in the registry base → redeploy → projects update with `npx shadcn add @unitbox/base --overwrite`.

## Ingestion Pipeline

How new design patterns enter the system:

```
1. User feeds a DOM dump / screenshot to Claude
2. Claude (via unitbox-ds skill) extracts:
   ├── New tokens → design-tokens.md + registry base cssVars
   ├── New components → component-rules.md + registry components/
   ├── New page patterns → page-patterns.md + registry blocks/
   ├── Anti-patterns → anti-patterns.md
   └── Log entry → ingestion-log.md (source URL, date, what extracted)
3. shadcn build + deploy registry
4. Existing projects: npx shadcn add @unitbox/base --overwrite (optional)
```

**ingestion-log.md format:**
```markdown
## 2026-04-03 — ElevenLabs App (27 dumps)
- Source: elevenlabs.io/app/* pages
- Extracted: design tokens, 20 component overrides, 8 page patterns, 20 micro-components
- Key findings: alpha-based borders, near-black primary, differentiated radius scale

## 2026-04-04 — ElevenLabs Landing
- Source: elevenlabs.io (homepage)
- Extracted: hero section pattern, feature card grid, code example block, landing typography scale
- Key findings: Waldenburg headings (weight 300), larger body text (16-18px), pill CTAs
```

## Team Sharing

### Skill (knowledge) — via git

All team members sync `~/.claude/` through git:
```bash
cd ~/.claude && git pull  # gets latest skills, memory, agents
```

Every Claude Code session reads `~/.claude/skills/unitbox-ds/` automatically.

### Registry (code) — via shadcn namespace

All projects point to the same registry URL in `components.json`. Install/update with standard shadcn CLI.

### Per-project rules — via CLAUDE.md

Each project can have additional rules in `.claude/CLAUDE.md`:
```markdown
# Nexa CRM — Project Rules
- Primary color is blue (--primary override in globals.css)
- This project uses Russian locale by default
- Table rows are clickable → always add cursor-pointer
```

## Cross-Project Override Model

```
@unitbox/base (default values)
│
├─→ Unitbox HQ     — uses defaults (near-black primary)
├─→ Nexa CRM       — overrides: --primary (blue), --radius-field (8px)
├─→ Catalog         — overrides: --primary (amber)
└─→ New Project X   — starts with defaults, customizes as needed
```

**What's shared:** All components, page patterns, anti-patterns, composition rules.
**What's per-project:** Colors, radius scale, fonts (if needed), project-specific CLAUDE.md rules.

## Hooks (Automation)

```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "if": "Write(components/ui/**)|Edit(components/ui/**)",
      "hooks": [{
        "type": "command",
        "command": "echo 'Verify: uses semantic tokens (rounded-field, not rounded-xl)'"
      }]
    }]
  }
}
```

## Implementation Phases

### Phase 1: Registry Scaffold (Day 1)
- Create `unitbox-registry/` from shadcn registry-template
- Define `registry:base` with all CSS variable tokens from current `globals.css`
- Define `registry:font` for Inter
- Deploy to Vercel
- Test: `npx shadcn add @unitbox/base` in a fresh Next.js project

### Phase 2: Component Migration (Day 2)
- Move overridden components (button, input, card, select, badge, dialog, etc.) into registry as `registry:ui` items
- Each component uses semantic tokens (rounded-field, not hardcoded values)
- Test: full install in clean project produces working UI

### Phase 3: Composed Blocks (Day 3)
- Create `registry:block` items for page patterns: data-table-page, settings-page
- Create `registry:component` items for composed widgets: filter-bar, status-badge
- Test: `npx shadcn add @unitbox/data-table-page` scaffolds a working page

### Phase 4: Skill + Ingestion (Day 4)
- Create `unitbox-ds` skill (rename from elevenlabs-style, make project-agnostic)
- Add `parse-dump.py` script for automated extraction
- Add `audit-tokens.sh` for skill ↔ registry alignment check
- Process the new landing page dump through the pipeline
- Update ingestion-log.md

### Phase 5: Team Onboarding (Day 5)
- Document the setup process
- Add `@unitbox` registry to all existing projects
- Verify all team members have the skill synced
- Add hooks for auto-audit
